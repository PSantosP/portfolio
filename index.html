CASE 01

반복 적용되는 광역 스킬의 데미지가 매 틱 1로 붕괴하던 참조 공유(aliasing) 버그

참조 타입 데미지 패킷을 매 틱 재사용하는 틱 스킬에서, 내부의 in-place 감산이 호출처 캐시를 영구 오염시켜 데미지가 1로 붕괴.

단발 스킬은 멀쩡한데 틱 스킬만 터진다 — 차이는 패킷 객체의 수명이었다. 틱 스킬은 GC를 아끼려 하나의 패킷을 캐시해 재사용하는데, 그건 참조 타입이라 피격 처리부의 방어 감산이 캐시 객체 자체를 깎는다(aliasing). 개별 처리부를 각각 고치는 대신, 모든 피격이 지나는 단일 진입점에서 원본을 스냅샷하고 finally로 원복해 불변식을 강제했다.

Before · 문제
// 피격 처리 진입점 — 각 캐릭터가 직접 구현(가상)
abstract bool ApplyHit(DamagePacket p);

// 내부에서 방어·실드 계산으로 p.Amount 를 그 자리에서 감산
// 틱 스킬은 하나의 패킷을 매 틱 재사용(참조 타입)
//  → 감산이 캐시에 누적: 100→40→16→ … →1(붕괴)
After · 수정
// 모든 피격이 지나는 단일 진입점(비가상 래퍼)
bool ApplyHit(DamagePacket p)
{
    var original = p.Amount;        // 값 스냅샷(할당 0)
    try     { return ApplyHitCore(p); } // 내부는 자유 변형
    finally { p.Amount = original; }  // 항상 원복
}
abstract bool ApplyHitCore(DamagePacket p);
  • 근본 vs 증상 — 호출처마다 복제(GC 폭증)가 아니라 진입점 한 곳에서 불변식을 강제 → 이후 모든 틱 스킬이 자동으로 안전.
  • 제로 GC — 복원을 위해 값 하나만 스냅샷, 모바일 GC 예산 존중.
  • 계약 명문화 — 내부 처리부에 "값을 변형해도 됨"을 문서화해 미래 구현자가 오염을 걱정할 필요 없음.

※ 실제 클래스/필드명은 익명화 · 구조 동일

CASE 02

궁극기 평타 버프의 StackOverflow — 무한 재귀를 다형성으로 구조 제거

버프 '배율'을 반환해야 할 함수가 '최종 데미지 계산'을 반환해 데미지 파이프라인을 재진입 → 특정 궁극기 발동 후 평타 시 무한 재귀 크래시.

유사 궁극기들이 멀쩡했던 건 발동 중 평타가 잠겨 버그 경로에 진입조차 못 했기 때문 — 버그는 전부에 있었다. 한 궁극기만 발동 중 공격이 가능해 드러났다. 해결은 재귀를 '고치기'가 아니라 '불가능하게' 만드는 것: 버프 계산을 각 궁극기가 소유하되 순수 배율만 허용(파이프라인 재진입 금지)하는 다형 계약으로 전환했다.

Before · 문제
// 특정 궁극기 — 배율 대신 '최종 데미지 계산'을 반환
double GetAttackBuff() {
    if (inWindow) return ComputeFinalDamage(); // ★재귀근원
    return 1;
}
// 평타 계산 — 종류를 문자열로 분기해 위를 호출
dmg *= container.GetAttackBuff(type);
// → GetAttackBuff → ComputeFinalDamage → (버프 재조회) → ∞
After · 수정
// 궁극기 베이스 — 순수 배율 계약(재진입 금지)
virtual double ModifyDamage(double baseDmg)
    => providesBuff && inWindow
        ? baseDmg * (1 + rate)   // 데이터만 사용
        : baseDmg;

// 개별 궁극기 — 의도만 선언
override bool providesBuff => true;

// 평타 계산 — 문자열 분기 제거, 위임 한 줄
dmg = container.ModifyDamage(dmg);
  • 재발 불가 설계 — 계약을 '데이터만 쓰는 순수 배율'로 좁히면 파이프라인 재진입이 곧 계약 위반, 같은 버그가 재발할 수 없음.
  • OCP(개방-폐쇄) — 신규 궁극기는 의도 플래그만 override, 중앙 코드 0 수정 (과거엔 문자열 분기·매핑 수정 필요).
  • 부수 효과 — 이전 dead path였던 버프가 실제 작동 + 코드 55줄 순감.

※ 실제 클래스/메서드명은 익명화 · 구조 동일 · 설계 문서 동반

CASE 03

라이브 패치를 반복할수록 번들 메모리가 누적되던 핸들 누수

에셋 다운로드 핸들이 완료·취소·예외 어느 경로에서도 해제되지 않아, 패치 사이클마다 번들이 메모리에 영구 누적되고 카탈로그 스왑 충돌을 유발.

메모리에 남은 번들은 패치 때마다 하나씩 늘었다. 진행률 폴링 루프의 탈출 경로(취소 return·정상 완료·예외) 어디에서도 Release를 부르지 않은 게 원인. 결정적 단서는 형제 API는 이미 해제 중이었다는 것 — 규칙은 있었고 이 경로만 빠진 일관성 결함이었다. try/finally로 전 경로 해제를 보장하고, 다음에 또 안 겪도록 진단 로그까지 붙였다.

Before · 문제
var handle = LoadBundlesAsync(labels);
while (!handle.IsDone)
{
    if (cancelled) return;   // 해제 없이 이탈
    report(progress);
    await Yield();
}
// 완료해도 Release 없음
// → 번들 영구 잔류: 패치마다 누적 + 스왑 충돌
After · 수정
var handle = LoadBundlesAsync(labels);
try
{
    while (!handle.IsDone)
    {
        if (cancelled) return;   // finally가 보장
        await Yield();
    }
}
finally
{
    // 완료·취소·예외 전 경로 해제(디스크 캐시 유지)
    if (handle.IsValid()) Release(handle);
}
  • 일관성 관점의 디버깅 — "형제 API는 지키는데 이 경로만 안 지킨다"는 패턴 인식으로 근본을 특정, 새 방식 발명이 아닌 계약 복원.
  • 재다운로드 아닌 재로드 — 핸들만 해제하고 디스크 캐시는 유지 → 사용자 요금·대기 없이 메모리만 회수.
  • 관측성까지 — 진단 로그를 해시 번들명 → 대표 에셋 경로로 바꿔 다음 누수의 출처를 즉시 식별.

※ 엔진 공개 API 외 실제 식별자는 익명화 · 구조 동일
